# RFC 0002 — KSF 1.0 stabilization

- Status: accepted
- Author(s): Kitbash maintainers
- Tracking PR: #18
- Depends on: [RFC 0001](0001-ksf.md)

## Summary

Freeze the KSF manifest surface so authors and third-party implementers can build against a stable contract. This RFC enumerates every `skill.toml` field, marks each **stable** (frozen at 1.0) or **provisional** (may still change), pins the artifact-schema versioning rule, and defines what "KSF 1.0" means and how it changes afterward. Ratifying it is the gate to dropping the "early preview" label.

## Motivation

Today the README says "the spec may change before 1.0," which is honest but unbuildable-against: nobody wants to author skills or write an adapter on sand. Ponytail earned trust with frozen behavior and versioned releases; KSF needs the equivalent — a line past which a field's meaning will not change under you. The manifest has been stable in practice across 0.1–0.3 (no field removed, only `dependencies` added). This RFC makes that stability a promise.

## Design

### Compatibility policy (post-1.0)

- **Additive only within a major.** New optional fields and new enum values may appear in minor releases. Existing fields never change meaning, type, or default within a major.
- **Removals and semantic changes require a major** (KSF 2.0) and a migration note.
- **Unknown fields are ignored with a lint warning**, never a hard error — forward compatibility for skills authored against a newer minor.
- **A skill declares nothing about which KSF version it targets.** The compiler is versioned; the format is designed so old skills compile on new compilers unchanged.

### Field status

**`[skill]`** — **stable.** `name`, `version` (semver), `description` required; `license`, `authors`, `homepage` optional. Frozen.

**`[context]`** — **stable.** `budget` (required), `standing`, `disclosure` (`lazy`|`eager`). The token-accounting core; frozen. *Note:* token counts are estimates (§ tokenizer note) — the field contract is frozen, the estimator may improve.

**`[permissions]`** — **stable shape, provisional vocabulary.** `tools` (string array), `network`, `write` frozen as fields. The *grammar* of a tool string (`"bash:git *"`) is provisional — a future minor may formalize it. Authors should treat tool strings as advisory today.

**`[artifacts]`** — **stable.** `produces` / `consumes` as `name@version` refs. Frozen. The *set* of first-party artifact schemas (`plan@1`, `findings@1`, `verify@1`, `benchmark@1`) versions independently under `spec/schema/artifacts/` (see below).

**`[targets]`** — **stable.** `requires` (capability list), `mode` (`skill`|`gate`). Frozen. New capability strings may be added additively.

**`[triggers]`** — **stable.** `commands`, `auto`, `events`. Frozen. Adapters emit what each host supports; unsupported triggers degrade visibly.

**`[dependencies]`** — **provisional.** Semver ranges between skills; resolution and lockfile pinning are implemented, but transitive-conflict resolution and the private-index story are unproven at scale. May gain fields before freezing.

**`[lore]`** — **provisional / experimental.** `reads` / `writes` collection names. Lore is explicitly separable (may ship as `@kitbash/lore`); its manifest surface is not frozen and may move out of core KSF entirely.

### Artifact schema versioning

Artifact types version in their `@N` suffix, independently of KSF and the CLI. A `plan@1` consumer must accept any `plan@1.x` document (additive fields only); a breaking change mints `plan@2`. Schemas live in `spec/schema/artifacts/` and each freezes on first stable use.

### What "KSF 1.0" ships

1. Every **stable** field above, frozen.
2. The conformance suite (from RFC 0001 §9): budget enforcement, standing-stub limits, degradation visibility, generated-file headers, permission honesty.
3. This compatibility policy, in the spec.
4. Provisional fields clearly marked in `spec/SPEC.md` so authors know what not to lean on.

Provisional fields (`dependencies` detail, `lore`, permission-string grammar) may reach 1.0 as-is (frozen) or stay marked provisional past 1.0 — a provisional field in a 1.0 spec is allowed, as long as it's labeled.

## Drawbacks

Freezing early risks locking a suboptimal field. Mitigated by: the additive-only policy leaves room to grow, and the genuinely unsettled parts stay explicitly provisional rather than being frozen prematurely.

## Alternatives

- **Freeze nothing until everything is proven.** Rejected: "may change" indefinitely is why nobody builds on a format. Stability is the product.
- **Freeze the entire manifest including lore/dependencies.** Rejected: those are genuinely unproven; freezing them now would force a KSF 2.0 sooner.

## Unresolved questions

- Tokenizer-specific budget accounting: does the frozen `budget` contract need a declared tokenizer, or does "estimate + generous default" suffice for 1.0?
- Permission-string grammar: formalize before or after 1.0?
- Does `lore` belong in KSF at all, or ship entirely as an extension namespace?
