# RFC 0001 — Kitbash Skill Format (KSF)

- Status: accepted (draft spec)
- Author(s): Kitbash maintainers
- Spec: [spec/SPEC.md](../spec/SPEC.md)

## Summary

An open, assistant-agnostic format for AI agent skills: a directory with a `skill.toml` manifest and `SKILL.md` instructions, plus optional scripts and evals. Compiled by adapters to each assistant's native format.

## Motivation

Every assistant invented its own extension format; skills are locked to one vendor, untested, unversioned, and unmeasured. See [MANIFESTO.md](../MANIFESTO.md).

## Design

The normative text lives in [spec/SPEC.md](../spec/SPEC.md), which this RFC ratifies as the v0.1 draft. Key decisions ratified here:

- Manifest declares **context budgets** (enforced at compile), **permissions** (honest enforcement only), **typed artifacts** (produces/consumes), **capability requirements** (visible degradation), and **dependencies** (semver, closure-pinned, no permission inheritance).
- KSF has its own identity: independent implementations are encouraged, conformance is defined by the spec's test suite (v1.0), and the spec moves to its own repository at 1.0.

## Drawbacks

A format nobody adopts is dead weight; the reference compiler must be good enough that the format spreads with it.

## Alternatives

- Extending AGENTS.md: text-only floor, no budgets/permissions/tests — structurally can't carry the requirements.
- Adopting one vendor's format as the standard: vendor-governed dialects don't become open standards.

## Unresolved questions

Tracked as issues: standing-stub measurement across tokenizers; artifact schema evolution rules; script sandboxing baseline.
