# Kitbash Skill Format (KSF) — spec draft v0.1

Status: **draft**. Everything here may change before 1.0. Changes land via [RFCs](../rfcs/README.md).

KSF is designed to outlive its reference implementation. Independent implementations are encouraged — "supports KSF" is the goal, not "requires Kitbash." Conformance is defined in §9; at spec 1.0, KSF moves to its own repository under open governance.

## 1. Skill layout

A skill is a directory:

```
<name>/
  skill.toml        # REQUIRED — manifest
  SKILL.md          # REQUIRED — instructions
  prompts/          # optional — named prompt templates referenced from SKILL.md
  scripts/          # optional — deterministic helpers (any executable)
  evals/            # optional, strongly encouraged — *.eval.yaml
  fixtures/         # optional — fixture material for evals
```

Names: `[a-z][a-z0-9-]{1,40}`.

## 2. Manifest — `skill.toml`

```toml
[skill]
name = "prereview"
version = "0.3.1"              # semver
description = "Review a diff against this team's real standards"
license = "Apache-2.0"
authors = ["Jane Dev <jane@example.com>"]
homepage = "https://github.com/kitbash-dev/prereview"

[triggers]
commands = ["/prereview"]      # explicit invocations to register
auto = []                      # natural-language trigger hints, adapter-dependent
events = ["pre-push"]          # hook points, for gate-mode skills

[context]
budget = 1500                  # REQUIRED. max tokens injected when active
standing = 80                  # max tokens present when NOT invoked (frontmatter/trigger stub)
disclosure = "lazy"            # lazy | eager

[permissions]
tools = ["read", "grep", "bash:git *", "bash:gh pr view *"]
network = false
write = false                  # may the skill direct file modifications?

[artifacts]
produces = ["findings@1"]
consumes = ["plan@1"]          # optional inputs; skill must work without them

[targets]
requires = []                  # capabilities: "scripts" | "hooks" | "subagents" | "network"
mode = "skill"                 # skill | gate

[lore]
reads = ["conventions", "invariants"]
writes = ["conventions"]       # only via human-reviewed proposals

[dependencies]
plan = "^1.0"                  # other skills this one builds on, semver ranges
```

Required tables: `[skill]`, `[context]`. Everything else defaults to the most restrictive value.

### Semantics that adapters MUST honor

- **`context.budget`** — compiled output for the active state must fit; compilers fail the build otherwise.
- **`context.standing`** — the always-loaded stub (trigger description) must fit; this is what keeps 30 installed skills from silently eating the context window.
- **`permissions`** — enforced natively where the platform supports it; otherwise compiled into the instructions and surfaced at install review. Adapters MUST NOT claim enforcement they don't have.
- **`targets.requires`** — a compiler lacking a required capability MUST either emit a degraded variant with a visible warning, or fail under `--strict`. Silent degradation is a conformance violation.

## 3. Instructions — `SKILL.md`

Plain markdown. First paragraph = the standing stub (what the skill does, when to invoke it) and must fit `context.standing`. The rest loads on invocation.

Template variables in double braces resolve at compile or invocation time:

- `{{lore.<collection>}}` — lore entries the manifest declared in `[lore] reads`
- `{{artifact.<type>}}` — path of a consumed artifact if present
- `{{prompt.<name>}}` — inclusion of `prompts/<name>.md`

Unknown variables are a lint error.

## 4. Scripts

- Never executed at install time. Only when the agent invokes the skill, within declared permissions.
- Must be idempotent or clearly named otherwise (`apply-*` vs `check-*`).
- Gate-mode skills MUST route their verdict through a script exit code or a schema-validated artifact — a bare LLM opinion is not a gate verdict.

## 5. Artifacts

JSON files under `.kitbash/artifacts/`, named `<type>.json` or `<type>/<id>.json`. Each type has a versioned schema in `spec/schema/artifacts/`. v0.1 reserves: `plan@1`, `findings@1`, `verify@1`, `benchmark@1`.

## 6. Evals — `*.eval.yaml`

```yaml
tier: behavioral            # static | audit | behavioral
fixture: fixtures/express-app
task: "prereview the diff on branch eval/sqli"
runner: claude-code         # headless runner; others as they expose -p equivalents
assert:
  - artifact: findings@1
    where: { file: "src/db.ts" }
    expect: { severity: high }
  - files_unchanged: ["src/**"]
```

Static-tier checks (schema, budgets, dead refs, injection heuristics) need no eval file; `kitbash test` always runs them.

## 7. Compilation model

`kitbash compile` resolves installed skills against detected adapters and emits native outputs. Outputs are build artifacts: deterministic given (skill content hash, adapter version, config). A `# generated by kitbash — do not edit` header is mandatory in every emitted file.

## 8. Versioning & distribution

- Skills: semver. Instruction changes that alter behavior are minor at least; permission or budget increases are major. Treat prompts like APIs: a breaking prompt change is a breaking change.
- Installs pin to a content hash in `kitbash.lock`; version tags are human conveniences, hashes are the truth.
- Source forms: `gh:owner/repo[/path][@ref]`, `https://` tarball, `file:`, index short name.

### 8.1 Dependencies

Skills may depend on other skills via `[dependencies]` (semver ranges). The resolver installs the transitive closure and pins every node in `kitbash.lock`. Cycles are an install-time error. A dependency's `produces` artifacts become available to the dependent's `consumes`. Dependents inherit nothing else — permissions and budgets never flow across the edge; each skill's manifest stands alone.

### 8.2 Compatibility

A skill's compatibility matrix is **derived, never hand-written**: `targets.requires` crossed with each adapter's capability set yields `full | degraded | unsupported` per assistant. `kitbash doctor` prints it locally; the index displays it per skill. Publishing a hand-claimed matrix is not a thing — the compiler is the source of truth.

## 9. Conformance

An adapter is conformant when it passes the conformance suite (ships with v1.0): budget enforcement, standing-stub limits, degradation visibility, generated-file headers, permission honesty.
