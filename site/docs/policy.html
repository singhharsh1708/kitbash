Trust & policy

Installing a skill means letting someone else's instructions run with your agent's permissions. Kitbash makes that reviewable — and, for teams, governable.

The review gate

Every kitbash install prints a review block before writing anything: permissions (including network and write access), token budget and standing cost, capability requirements, and lint findings including injection heuristics. On a TTY it then prompts:

review: code-review-and-quality@0.0.0 — Conducts multi-axis code review…
  budget 6000 tokens · standing 250 · lazy disclosure · mode skill
  permissions: tools [none] · network no · write no
  ⚠ unmanifested (SKILL.md only) — defaults applied
install? [y/N]

--yes/-y skips the prompt (CI runs proceed as before) — but policy, below, is never skipped.

Org policy

In kitbash.toml:

[policy]
allow_sources = ["gh:your-org/*"]  # globs matched against gh:owner/repo[/path][@ref] or file: sources
deny_network = true                # refuse skills declaring network permission
deny_write = true                  # refuse skills declaring write permission
max_budget = 6000                  # refuse skills with a larger context budget
  • Policy is a hard gate at install — --yes does not bypass it.
  • kitbash doctor rechecks policy against already-installed skills, so tightening the rules flags existing violations.

Integrity

  • Installs pin a sha256 content hash over the skill directory in kitbash.lock (CRLF/LF-insensitive, path-order stable).
  • doctor exits 1 when installed files drift from the lock — instructions cannot change on disk unnoticed.
  • Every emitted file carries a generated by kitbash header, and only marked files are ever rewritten or pruned. Shared files (AGENTS.md, GEMINI.md, CONVENTIONS.md) are marker-merged — your own content is untouched.

Read before you install

preview, lint, and explain all accept uninstalled sources — kitbash preview gh:owner/repo/path fetches to a temp directory and renders the exact compiled output per agent, with no side effects.
